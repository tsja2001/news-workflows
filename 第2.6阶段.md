# Phase 2.6:深度抓取 + 完整审计日志

> 本任务书追加在 Phase 2.5 之后,Phase 3 之前。
> 在 Phase 2.5 已合并、人类 review 通过后启动。
> 全局开发约束(代码风格、依赖管理、兼容性、测试要求、提交规范、不要做的事)沿用 PLAN.md。

---

## 背景

Phase 2.5 完成了 `type: web` 通用 AI 抓取。但实际使用中暴露两个问题:

**问题 1:抓取广度不足**
- 一个 source 只能配一个 listUrl,而很多新闻站有多个版块(BBC 中东 / 世界 / 政治),想要全部就得拆成多个 source
- 大多数新闻站列表页有分页(?page=N 或 ?offset=N),当前只抓第一页
- AI 提取时倾向于"识别出明显的几条",遗漏侧栏推荐、专题区、时间较老的条目
- `maxArticles` 默认 10 条,对"深度抓取"场景太保守

**问题 2:抓取过程是黑盒**
- 终端日志只显示"提取到 12 条",但具体是哪 12 条、为什么不是 50 条、列表页一共有多少候选、最终多少进了简报——无从知晓
- 没法事后审计:某条本该出现的新闻为什么没出现?是没抓到、还是被过滤了、还是历史去重剔除了?
- 抓取失败时只看到 error,看不到当时 AI 看到了什么、做了什么判断

## Phase 2.6 目标

针对上面两个问题,做两件事:

1. **深度抓取**:支持多 listUrl、URL 分页、AI 穷尽提取、放开数量上限
2. **完整审计**:每次抓取生成 JSONL 审计日志,记录从"AI 看到的所有候选链接"到"最终进入简报的条目"的完整流转

明确的取舍:
- **不在意 token 消耗**(用户已确认)
- **不在意单次抓取耗时**(用户已确认)
- **不做** AI 临场决策的复杂交互(点击"加载更多"、跟进相关链接等)
- **不做** 跨 source 的智能合并/去重策略(过滤管线已有 URL 去重)

## 设计原则(本 Phase 特有)

1. **配置兼容**:Phase 2.5 已有的 `url` 字段继续工作,新功能纯叠加
2. **审计日志独立**:不替换 logger,审计日志写到文件,终端日志保持简洁
3. **AI 信任但可追溯**:让 AI 大胆提取,但每一步决策都记录,事后可复查
4. **数据可查询**:JSONL 格式,方便用 `jq` 等工具查询

# Phase 2.6.1:多 listUrl + 分页支持

**目标**:让一个 source 能覆盖多个版块、多页内容,而不需要拆成多个 source。
**Branch**:`feature/phase-2.6.1-multi-list`

## 1.A 多 listUrl 支持

### yaml schema 扩展

```yaml
# 老格式(继续支持)
- name: BBC 中东
  type: web
  url: "https://bbc.com/news/world/middle_east"

# 新格式 1:多 URL,每个独立配置
- name: BBC News(多版块)
  type: web
  urls:
    - "https://bbc.com/news/world/middle_east"
    - "https://bbc.com/news/world/asia"
    - "https://bbc.com/news/politics"
  maxArticles: 30   # 多 URL 共享这个上限,不是每个 URL 30 条

# 新格式 2:每个 URL 可以带独立 hint
- name: BBC News(细致版)
  type: web
  urls:
    - url: "https://bbc.com/news/world/middle_east"
      hint: "中东相关"
    - url: "https://bbc.com/news/world/asia"
      hint: "亚洲相关"
```

### 字段语义

- `url` (单数) 和 `urls` (数组) 是互斥的,二者必须有一个,不能都有
- 都没有时报错
- `urls` 数组里既支持纯字符串也支持对象 `{ url, hint }`
- 顶层的 `hint` 字段在 `urls` 模式下作为默认提示,被单 URL 的 hint 覆盖

### 实现要点

`src/fetch/web.js`:
- 增加 `expandUrls(source)` 函数,把 url/urls 标准化为统一的 `[{ url, hint }]` 数组
- 主流程改为对每个 URL 独立处理(渲染、提取列表),最后合并成一个大列表
- 多个 URL 之间在 source 内部串行处理(不要并发,会撑爆浏览器内存)
- 每个 URL 失败不影响其他 URL,但全部失败时整个 source 失败

```js
// 伪代码
async function fetchFromWeb(source) {
  const expandedUrls = expandUrls(source);
  const allCandidates = [];

  for (const { url, hint } of expandedUrls) {
    try {
      const candidates = await fetchOneListPage(url, hint, source);
      allCandidates.push(...candidates);
    } catch (err) {
      // 失败的 URL 记录但不中断
      logger.warn('列表页失败,跳过', { url, error: err.message });
    }
  }

  if (allCandidates.length === 0) {
    throw new Error('所有列表 URL 均失败或为空');
  }

  // URL 去重(同一篇文章可能在多个版块出现)
  const deduped = dedupByUrl(allCandidates);

  // 截取前 maxArticles 条
  const selected = deduped.slice(0, source.maxArticles ?? 10);

  // 抓取详情(沿用 Phase 2.5 逻辑)
  ...
}
```

## 1.B URL 分页支持

### yaml schema 扩展

```yaml
# 模板分页
- name: 某站
  type: web
  url: "https://example.com/news?page={page}"
  pages: 3                  # 抓 page=1, page=2, page=3
  pageStart: 1              # 默认 1,有些站从 0 开始

# 也可以用在 urls 数组里
- name: 多版块带分页
  type: web
  urls:
    - url: "https://example.com/world?p={page}"
      pages: 2
    - url: "https://example.com/politics?p={page}"
      pages: 3
    - "https://example.com/single"   # 这个不分页
```

### 字段语义

- `{page}` 是固定占位符,会被替换成数字
- `pages` 不存在时按单页处理(不展开)
- 没有 `{page}` 但有 `pages` 时:报错(配置错误)
- 有 `{page}` 但没 `pages` 时:报错
- 翻页之间在浏览器层面**复用 context** 但**新建 page**(避免 SPA 路由污染)

### 实现要点

`src/fetch/web/url-expander.js` 新模块:

```js
/**
 * 把 source 的 url/urls + pages 配置展开为最终要抓的 URL 列表
 * @returns {Array<{ url: string, hint: string, page?: number }>}
 */
export function expandUrls(source) { ... }
```

例:
```yaml
urls:
  - url: "https://a.com/?p={page}"
    pages: 2
  - "https://b.com/news"
```
展开为:
```js
[
  { url: "https://a.com/?p=1", hint: "...", page: 1 },
  { url: "https://a.com/?p=2", hint: "...", page: 2 },
  { url: "https://b.com/news", hint: "...", page: null },
]
```

### 翻页之间的延迟

加可配置的页间延迟,避免触发反爬:

```yaml
- name: 某站
  type: web
  url: "https://example.com/news?page={page}"
  pages: 5
  pageDelayMs: 2000   # 默认 1000
```

### 总数上限保护

为防止配置失误(比如 pages: 100)炸服务器:

- 全局硬上限:**单个 source 最多展开 20 个 URL**(包括多 listUrl × 多页)
- 超过时报错,提示用户检查配置

# Phase 2.6.2:AI 穷尽提取 + 放开数量

**目标**:让 AI 提取得更彻底,默认抓取上限放开,适配"不在意 token"的场景。
**Branch**:`feature/phase-2.6.2-deep-extract`
**前置依赖**:Phase 2.6.1 已合并

## 2.A 改造提取 prompt

### 当前问题

Phase 2.5 的 list 提取 prompt 措辞偏保守("识别出新闻条目"),AI 倾向于只列出"最显眼的"那部分,
遗漏侧栏专题、时间较老的条目、子分类列表等。

### 新 prompt(深度模式)

新增 `src/fetch/web/prompts.js` 里 `LIST_EXTRACT_DEEP_SYSTEM`:

```
你是一个网页内容提取助手,任务是从新闻列表页 HTML 中找出**所有**可能的新闻条目。

**请尽可能穷尽地列出**,包括:
- 主列表中的新闻
- 侧栏的"热门"、"推荐"、"编辑精选"
- 页面中段的专题区、深度报道
- 页脚之上的"更多新闻"、"相关阅读"区块
- 任何 <a> 标签指向新闻详情页的链接

返回严格的 JSON 数组,不要 markdown,不要解释:
[
  {
    "title": "新闻标题",
    "url": "新闻详情页 URL(相对或绝对)",
    "publishedAt": "ISO8601 时间或空字符串",
    "summary": "如果列表页有摘要则填,否则空字符串",
    "section": "这条新闻所在的版块/分类(如'头条'/'专题'/'侧栏推荐'/空字符串)",
    "confidence": "high | medium | low"
  }
]

confidence 判定:
- high: 明确是新闻条目,有标题和链接
- medium: 看起来像新闻但不确定(比如可能是专题页入口)
- low: 链接文字短或不像新闻标题,但属于内容区

**排除**:
- 顶部主导航、用户菜单、订阅按钮、登录链接
- 页脚版权、关于我们、联系方式
- 社交媒体分享按钮
- 广告、订阅推广

**不要去重、不要排序、不要筛选数量上限**——把所有候选都列出来,后续会自行处理。

**找不到合适条目时返回空数组 []**。
```

关键变化:
- 强调"穷尽"而非"识别"
- 引入 `confidence` 字段,让 AI 标记自己的不确定性,而不是直接丢弃可疑条目
- 引入 `section` 字段,后续审计可见
- 明确告诉 AI 不要自己限制数量

### user prompt 改造

```
URL: {currentUrl}
{hint ? `提示:${hint}` : ''}
{section ? `当前页面属于:${section}` : ''}

HTML(已瘦身):
{cleanedHtml}
```

### 切换机制

通过 yaml 配置选择提取深度:

```yaml
- name: 某站
  type: web
  url: ...
  extractDepth: deep   # 'normal' | 'deep',默认 'deep'
```

为什么默认 deep:用户已明确"不在意 token",且本 Phase 的核心目的就是深度抓取。

## 2.B 放开数量上限

### 默认值调整

| 配置 | 旧默认 | 新默认 | 说明 |
|------|-------|-------|------|
| `maxArticles` | 10 | 50 | 单个 source 最多抓多少条详情 |
| `detailConcurrency` | 2 | 3 | 详情页并发(轻微提升) |
| `filter.maxItems` | 40 | 80 | 过滤管线最终保留数 |

### 抓取数量逻辑

新增"候选池"概念:

```
所有 listUrl × 所有页 → AI 提取的所有候选(可能 200+)
        │
        ▼
URL 去重 + 协议过滤(http/https)
        │
        ▼
按 confidence 排序(high → medium → low)
        │
        ▼
取前 maxArticles 条进入详情抓取
        │
        ▼
详情成功的条目进入过滤管线
        │
        ▼
过滤管线 → maxItems 条 → 喂给 LLM
```

注意点:
- low confidence 的条目不丢弃,排在最后,凑齐 maxArticles 时才会用到
- 这样既"穷尽提取",又有合理的优先级
- 详情抓取阶段失败的条目不补,简单来说就是"宁可少不要补低质量"

### 全局硬上限保护

防止失误炸服务器:

- 单个 source 候选池最多保留 500 条(超出截断,记 warn)
- 单个 source 最多发起详情请求 100 条(`maxArticles` 上限,超出报错让用户改 yaml)

## 2.C 详情提取也要更"宽容"

### 当前问题

Phase 2.5 的 detail 提取在 readability 失败时降级到 ai,但没有针对"内容不完整"的情况进一步处理。

### 新增策略:深度提取模式

`detailExtraction: deep` 新模式,行为:

1. 先 readability 提取
2. 不论成功失败,**总是**再用 AI 提取一次
3. 取两者中**正文长度更长**的结果
4. 两个结果都记入审计日志

理由:有些新闻站 readability 能拿到正文但漏掉了开头摘要或尾部图说明,AI 能拿到完整版。
不在意 token 的前提下,两次都做有意义。

```yaml
- name: 某站
  type: web
  url: ...
  detailExtraction: deep   # 'auto' | 'readability' | 'ai' | 'deep'
```

默认仍是 `auto`(向后兼容),但建议在新源用 `deep`。

# Phase 2.6.3:审计日志(JSONL)

**目标**:每次抓取生成完整的 JSONL 审计日志,记录每个 source 看到的所有候选、最终选择、详情结果。
**Branch**:`feature/phase-2.6.3-audit-log`
**前置依赖**:Phase 2.6.2 已合并

## 3.A 设计目标

每次 `npm run brief <topic>` 运行,生成一份 JSONL 审计日志。
日志要回答以下问题:

1. **看了什么**:每个 source 访问了哪些 URL,每个 URL AI 提取出了多少候选?
2. **选了什么**:这些候选里哪些进入了详情抓取?为什么是这些?
3. **拿到了什么**:每个详情页抓取的成功/失败、用了哪种策略、字数多少?
4. **过滤了什么**:抓回来的条目里哪些被时间窗口/关键词/历史去重剔除了?
5. **简报里有什么**:最终喂给 LLM 多少条,是哪些?

## 3.B 文件位置和命名

```
logs/audit/
├── 2026-05-05/
│   ├── us-iran-2026-05-05-143052.jsonl       # 主审计日志
│   └── us-iran-2026-05-05-143052.summary.json # 一份汇总(便于快速查看)
└── 2026-05-06/
    └── ...
```

`.jsonl`:每行一个 JSON 对象,事件流式写入,失败也能看到部分日志
`.summary.json`:运行结束时生成的汇总,人类阅读用

## 3.C 事件 schema

每条审计日志都是一个事件,有统一信封:

```json
{
  "ts": "2026-05-05T14:30:52.123Z",
  "topic": "us-iran",
  "runId": "20260505-143052-abc12",
  "source": "BBC News",
  "sourceType": "web",
  "event": "list_extracted",
  "data": { ... }
}
```

`runId` 是本次运行的唯一标识,所有事件共享。

### 关键事件类型

| event | 触发时机 | data 字段 |
|-------|---------|----------|
| `run_started` | 流程启动 | `{ topic, sourcesPlanned: 12 }` |
| `source_started` | 单个 source 开始 | `{ urls: [...], maxArticles: 50 }` |
| `list_page_loaded` | 列表页渲染完成 | `{ url, page?, htmlBytes, durationMs }` |
| `list_extracted` | AI 提取列表完成 | `{ url, page?, candidates: [...], tokens, durationMs }` |
| `candidates_filtered` | 候选去重 + 排序完成 | `{ before, after, dropped: [...] }` |
| `detail_started` | 单个详情页开始抓取 | `{ url, title }` |
| `detail_extracted` | 详情抓取成功 | `{ url, strategy, length, tokens?, durationMs }` |
| `detail_failed` | 详情抓取失败 | `{ url, reason, durationMs }` |
| `source_completed` | source 处理完成 | `{ totalCandidates, detailsFetched, detailsSucceeded, durationMs }` |
| `source_failed` | source 整体失败 | `{ reason, durationMs }` |
| `pipeline_filter` | 过滤管线每一步 | `{ stage, before, after, dropped: [...] }` |
| `llm_input_prepared` | 发给 LLM 的最终列表 | `{ itemCount, items: [...] }` |
| `llm_response_received` | LLM 返回 | `{ tokens, durationMs }` |
| `run_completed` | 全流程完成 | `{ totalSources, succeeded, failed, durationMs }` |

### 关键 data 字段约束

**`list_extracted` 的 candidates 字段必须完整**:

```json
{
  "event": "list_extracted",
  "data": {
    "url": "https://bbc.com/news/world/middle_east",
    "candidates": [
      {
        "title": "...",
        "url": "https://bbc.com/news/...",
        "publishedAt": "...",
        "summary": "...",
        "section": "头条",
        "confidence": "high"
      },
      // ... 完整列表,不限制条数
    ],
    "tokens": { "input": 4521, "output": 842 },
    "durationMs": 5234
  }
}
```

**`candidates_filtered` 的 dropped 字段说明剔除原因**:

```json
{
  "event": "candidates_filtered",
  "data": {
    "before": 87,
    "after": 50,
    "dropped": [
      { "url": "...", "reason": "url_invalid" },
      { "url": "...", "reason": "duplicate" },
      { "url": "...", "reason": "exceeded_maxArticles", "confidence": "low" },
      // ...
    ]
  }
}
```

**`pipeline_filter` 描述过滤管线每一步**:

```json

    ]

```

注意:这里**不写 content 全文**(JSONL 会过大),只写长度。原文已在 source URL,需要时可以重新抓。

## 3.D 实现:auditor 模块

### 模块:`src/utils/auditor.js`

```js
/**
 * 创建一个审计日志记录器
 * @param {object} runContext - { topic, runId, logDir }
 * @returns {Auditor}
 */
export function createAuditor(runContext) { ... }

// Auditor 接口:
auditor.event(eventType, data)        // 写一条事件
auditor.scoped(source, sourceType)    // 返回带 source 上下文的子 auditor
auditor.finalize()                    // 写 summary 文件并关闭
```

### 实现要点

1. **追加写**:用 `fs.createWriteStream(path, { flags: 'a' })`,流式追加
2. **同步刷盘**:每条事件 write 后调用 `flush`(或用 `fs.appendFileSync` 简单粗暴),确保崩溃时不丢
3. **目录自动创建**:`logs/audit/${date}/` 不存在自动 mkdir
4. **runId 生成**:`${YYYYMMDD-HHMMSS}-${5位随机字母}`
5. **scoped 子 auditor**:自动带上 source 字段,不需要每次手写
6. **finalize 时**:遍历整个 jsonl 文件,生成 summary.json(详见 3.E)

### 接入点

主要在以下位置插入审计调用:

| 位置 | 调用 |
|------|------|
| `src/index.js` 流程开始 | `auditor.event('run_started', ...)` |
| `src/index.js` 流程结束 | `auditor.event('run_completed', ...)` + `auditor.finalize()` |
| `src/fetch/index.js` 每个 source 开始 | `sourceAuditor.event('source_started', ...)` |
| `src/fetch/web.js` 列表页加载 | `sourceAuditor.event('list_page_loaded', ...)` |
| `src/fetch/web/extract-list.js` 提取完 | `sourceAuditor.event('list_extracted', ...)` |
| `src/fetch/web/extract-detail.js` 详情抓取 | `sourceAuditor.event('detail_extracted' | 'detail_failed', ...)` |
| `src/fetch/common.js` 每一步过滤 | `auditor.event('pipeline_filter', ...)` |
| `src/summarize.js` 调用前后 | `auditor.event('llm_input_prepared', ...)` 和 `'llm_response_received'` |

其他 adapter(rss/html/api/playwright)也要接入相应的事件,但不强求和 web 一样详细——
最低要求:`source_started`、`source_completed/failed`、对每条最终输出的条目记录。

- 不写 prompt 全文

- 单条事件大小 ≤ 100KB(防止某次 AI 返回异常大数据撑爆日志)

# 给 codex 的特别提醒(Phase 2.6)

## 容易踩的坑

1. **多 URL 时 maxArticles 语义混淆** → 明确:是"整个 source 的总上限",不是每个 URL 的上限
2. **分页 {page} 占位符没替换** → 必须先校验,有 {page} 必须有 pages
3. **审计日志写满磁盘** → prune 命令必须可靠,默认保留期不要设太长
4. **审计日志出现循环引用 / 不可序列化对象** → JSON.stringify 前必须 sanitize
5. **多个 source 并发写同一个 jsonl** → 用 appendFileSync 或加文件锁,不要用 createWriteStream(并发不安全)
6. **AI 返回的 candidates 太多撑爆终端** → 终端只显示前 3 + 总数,完整列表在审计日志
7. **summary 计算时遍历整个 jsonl** → 大日志可能慢,但接受这个慢(运行结束才算,不影响主流程)
8. **deep 模式的两次 detail 提取消耗双倍 token** → 文档里明确告知用户

## 何时停下来问人类

- AI prompt 措辞改了之后效果(深度模式效果好不好,需要看实际抓取量)
- 终端日志的视觉密度(候选分组打不打、打多少行)
- audit CLI 的命令名和输出格式
- prune 的默认保留期(我给了 30 天,但你可能想调整)
- summary 里成本估算的展示形式
