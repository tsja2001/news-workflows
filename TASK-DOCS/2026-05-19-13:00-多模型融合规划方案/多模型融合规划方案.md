# 新闻简报多模型融合规划方案

## 1. 背景与目标

当前新闻简报流水线是单模型架构：抓取和过滤后的全部新闻条目一次性送入 `src/summarize.js`，再由 `src/llm.js` 调用 `.env` 中配置的 DeepSeek 兼容接口，直接生成最终 Markdown 所需的结构化 JSON。

这个架构简单，但问题也集中：

- 抓取后的新闻数据量大，直接把全量素材交给高价模型会显著增加成本。
- DeepSeek 适合做批量信息处理、结构化整理、初步筛选，但在最终观点表达、选题判断、文章气质和主编复盘上不够稳。
- 当前只有一个全局 `LLM_MODEL`，无法按任务类型选择不同模型，也无法在审计日志中拆分不同阶段的成本。

本次优化目标是引入多模型融合：**DeepSeek 负责大批量低成本处理，Claude 负责高价值最终判断和成文**。

具体目标：

- 控制 Claude 输入规模，只让它读取经过压缩、聚类、排序后的高价值素材。
- 保留 Claude 对最终选题、观点判断和文字表达的主导权。
- 保持现有 CLI、主题配置和输出结构尽量兼容。
- 新增阶段级审计，能看清 DeepSeek 和 Claude 各自消耗多少 token、多少钱、耗时多久。
- 支持无 Claude 配置时自动回退到当前单模型模式。

## 2. 推荐架构

推荐采用“两段式主流程 + 可选校验阶段”。

```text
RSS / HTML / API / Playwright 抓取
  ↓
fetch/common.js 过滤、排除关键词、去重、截断
  ↓
DeepSeek 预处理阶段
  - 对大量原始 items 做压缩
  - 生成候选主线、事实要点、聚类、重要性初排
  - 标记低价值/应丢弃条目
  ↓
Claude 最终编辑阶段
  - 基于 DeepSeek 中间稿 + 必要原文证据
  - 重新判断 keyDevelopments
  - 写 TL;DR、overview、editorTake、editorReview
  - 输出现有 output.js 可消费的最终 JSON
  ↓
可选 DeepSeek 校验阶段（二期）
  - 检查 JSON schema、事实越界、排除话题、空字段
  ↓
output.js 写 Markdown
```

这个设计的关键点是：DeepSeek 不产最终文章，它只负责把大素材池变成 Claude 可以高效阅读的“编辑案头材料”。Claude 不需要读全部原始长文本，但必须拿到足够证据，避免只基于 DeepSeek 的二手判断写作。

## 3. 模型分工

### 3.1 DeepSeek：预处理模型

DeepSeek 适合承担以下任务：

- 大批量新闻条目的初步阅读和压缩。
- 按主题聚类，例如“中东航道”“俄乌谈判”“中美俄互动”“台海/南海”“制裁与供应链”。
- 每条候选主线提取事实要点、涉及主体、时间、来源 URL、可引用证据。
- 初步判断重要性：`highCandidate`、`mediumCandidate`、`briefOnly`、`drop`。
- 找出重复报道和弱相关素材。
- 输出结构化中间 JSON，而不是写成稿。

DeepSeek 输出应该克制、短、证据化，避免长篇主观表达。它的角色是“研究助理”，不是“主编”。

### 3.2 Claude：最终编辑模型

Claude 适合承担以下任务：

- 最终决定哪些主题进入 `keyDevelopments`。
- 判断排序：什么对读者最重要，而不是哪条新闻最新。
- 写每条关键变化的三段结构：`what`、`why`、`editorTake`。
- 写 `tldr`、`overview`、`signals`、`risks`、`unknowns`、`editorReview`。
- 根据 `editorial.persona`、`tone`、`interests` 和 `excludeTopics` 形成稳定的私人内参风格。

Claude 不应只做“润色”。选题判断本身就是核心编辑能力，应交给 Claude 完成。

## 4. 数据流设计

### 4.1 输入 items

现有 items 字段大致包括：

- `source`
- `publishedAt`
- `title`
- `summary`
- `url`
- `content`（部分 RSS 回抓正文时存在）

预处理阶段不应该无脑把完整 `content` 全塞给 DeepSeek。建议先在代码侧构造轻量输入：

```js
{
  id: 1,
  source,
  publishedAt,
  title,
  summary,
  contentExcerpt,
  url
}
```

其中 `contentExcerpt` 可以限制在每条 800-1200 字符，优先保留标题、摘要和正文开头。这样 DeepSeek 可以看足够上下文，但不会被全文成本拖垮。

### 4.2 DeepSeek 中间 JSON

建议新增中间结构：

```json
{
  "clusters": [
    {
      "clusterId": "middle-east-energy",
      "title": "中东航道与能源风险升温",
      "importanceHint": "high",
      "reason": "涉及能源通道、军事升级和市场外溢",
      "itemIds": [1, 4, 9],
      "facts": [
        {
          "itemId": 1,
          "fact": "事实要点，不评价",
          "source": "Reuters",
          "url": "https://..."
        }
      ],
      "whyItMayMatter": "结构性影响的初步说明",
      "openQuestions": ["后续需要确认的问题"]
    }
  ],
  "briefCandidates": [
    {
      "itemId": 12,
      "region": "欧洲",
      "summary": "一句话短讯素材",
      "reason": "相关但不足以展开"
    }
  ],
  "dropCandidates": [
    {
      "itemId": 20,
      "reason": "体育/娱乐/弱相关/重复报道"
    }
  ],
  "timelineCandidates": [
    {
      "time": "MM-DD HH:mm",
      "event": "事实事件",
      "itemId": 1
    }
  ]
}
```

注意：`clusters` 是给 Claude 的参考，不是最终答案。Claude 可以推翻 DeepSeek 的重要性排序。

### 4.3 Claude 输入

Claude 输入应包含三块：

1. `editorial` 配置：读者关心什么、不关心什么、语气和人设。
2. DeepSeek 中间 JSON：聚类、候选主线、事实要点、短讯候选、丢弃建议。
3. 精简原始素材包：只给进入 `clusters` 和 `briefCandidates` 的 item 原始标题、摘要、URL、必要摘录。

这样 Claude 仍能基于原始素材做判断，不会完全依赖 DeepSeek 的二手总结。

## 5. 配置设计

建议保留现有 `.env` 单模型变量作为兼容入口，同时新增多模型配置。

### 5.1 环境变量

```env
# 默认模型，兼容旧逻辑
LLM_API_KEY=...
LLM_BASE_URL=https://api.deepseek.com
LLM_MODEL=deepseek-v4-pro
LLM_TEMPERATURE=0.6

# 预处理模型：DeepSeek
LLM_PREPROCESS_API_KEY=...
LLM_PREPROCESS_BASE_URL=https://api.deepseek.com
LLM_PREPROCESS_MODEL=deepseek-v4-pro
LLM_PREPROCESS_TEMPERATURE=0.2
LLM_PREPROCESS_INPUT_PRICE_PER_1M_TOKENS=0.14
LLM_PREPROCESS_OUTPUT_PRICE_PER_1M_TOKENS=0.28

# 最终写作模型：Claude
LLM_WRITER_API_KEY=...
LLM_WRITER_BASE_URL=https://api.anthropic.com/v1
LLM_WRITER_MODEL=claude-3-7-sonnet-latest
LLM_WRITER_TEMPERATURE=0.7
LLM_WRITER_INPUT_PRICE_PER_1M_TOKENS=...
LLM_WRITER_OUTPUT_PRICE_PER_1M_TOKENS=...
```

如果通过 OpenAI 兼容网关调用 Claude，也可以只配置 `LLM_WRITER_BASE_URL` 为网关地址，继续复用 `ChatOpenAI`。如果直接调用 Anthropic 官方 API，需要新增 `@langchain/anthropic` 或 Anthropic SDK。

### 5.2 主题 YAML 配置

建议在 topic yaml 中增加可选段落：

```yaml
llmPipeline:
  mode: hybrid          # single | hybrid
  preprocess:
    provider: preprocess
    maxInputItems: 80
    itemExcerptChars: 1000
    maxClusters: 12
    maxFactsPerCluster: 5
  writer:
    provider: writer
    maxSourceItems: 35
    requireOriginalEvidence: true
  validation:
    enabled: false
    provider: preprocess
```

字段含义：

- `mode: single`：保持旧的一次性总结。
- `mode: hybrid`：启用 DeepSeek 预处理 + Claude 成稿。
- `maxInputItems`：预处理阶段最多读多少条。
- `itemExcerptChars`：每条正文摘录长度。
- `maxClusters`：DeepSeek 最多产出多少组候选主线。
- `maxSourceItems`：Claude 最多读取多少条精简原始素材。
- `requireOriginalEvidence`：Claude 阶段必须同时看到原始标题/摘要/URL。
- `validation.enabled`：是否启用校验阶段。

默认策略：如果未配置 `llmPipeline`，走现有单模型逻辑，确保向后兼容。

## 6. 代码改造规划

### 6.1 `src/llm.js`：从单模型封装改为多模型客户端

当前 `createModel()` 只读取 `LLM_*`。建议改成：

- `createModelClient(role)`：按 `default`、`preprocess`、`writer`、`validator` 读取不同环境变量。
- `callLLMForJsonWithMeta(systemPrompt, userPrompt, options)`：新增 `options.role`、`options.stage`。
- 返回值增加：
  - `role`
  - `stage`
  - `provider`
  - `model`
  - `tokens`
  - `rawTextLength`

兼容逻辑：

- 如果 `role=writer` 但没有 `LLM_WRITER_*`，回退到 `LLM_*`。
- 如果 `role=preprocess` 但没有 `LLM_PREPROCESS_*`，回退到 `LLM_*`。

### 6.2 新增 `src/summarize/preprocess.js`

职责：

- 把 items 转成受控长度的 `modelItems`。
- 构建 DeepSeek 预处理 prompt。
- 调用 `callLLMForJsonWithMeta(..., { role: 'preprocess', stage: 'preprocess' })`。
- 校验中间 JSON 的基本结构。
- 记录审计事件 `llm_stage_completed`。

### 6.3 新增 `src/summarize/write-report.js`

职责：

- 基于 DeepSeek 中间稿和精简原始素材构建 Claude prompt。
- 调用 `callLLMForJsonWithMeta(..., { role: 'writer', stage: 'final_write' })`。
- 输出与当前 `output.js` 完全兼容的 report JSON。

### 6.4 改造 `src/summarize.js`

建议让它变成编排层：

```js
export async function summarize(items, config, options = {}) {
  if (config.llmPipeline?.mode === 'hybrid') {
    const prep = await preprocessItems(items, config, options)
    return await writeFinalReport(items, prep.result, config, options)
  }

  return await summarizeSingleModel(items, config, options)
}
```

这样旧逻辑还在，新增逻辑清晰隔离。

### 6.5 `src/utils/auditor.js`：支持多阶段 LLM 成本

现有 summary 只有一个 `summary.llm`。建议扩展为：

```json
{
  "llm": {
    "stages": [
      {
        "stage": "preprocess",
        "role": "preprocess",
        "model": "deepseek-v4-pro",
        "inputTokens": 50000,
        "outputTokens": 6000,
        "estimatedCost": "约 ¥0.01",
        "durationMs": 12000
      },
      {
        "stage": "final_write",
        "role": "writer",
        "model": "claude-3-7-sonnet-latest",
        "inputTokens": 12000,
        "outputTokens": 5000,
        "estimatedCost": "约 ¥...",
        "durationMs": 18000
      }
    ],
    "totalInputTokens": 62000,
    "totalOutputTokens": 11000
  }
}
```

同时保留旧字段 `summary.llm.model/inputTokens/outputTokens`，避免 audit CLI 旧命令直接失效。

## 7. Prompt 设计原则

### 7.1 DeepSeek prompt

DeepSeek prompt 应强调：

- 不写成稿。
- 不写漂亮话。
- 不做最终结论。
- 每个判断必须绑定 `itemId` 和来源。
- 输出尽量短，给 Claude 降噪。

核心指令：

```text
你是新闻研究助理，不是主编。你的任务是把大量新闻素材整理成给主编使用的案头材料。
不要写最终简报，不要润色，不要扩写。
只输出 JSON。
每个事实必须关联 itemId。
对弱相关、重复、娱乐体育、与读者兴趣无关的素材做降级或丢弃建议。
```

### 7.2 Claude prompt

Claude prompt 应强调：

- 你是最终主编，可以推翻研究助理排序。
- 只基于给定素材，不编造。
- 选题和表达要对读者负责。
- `editorTake` 和 `editorReview` 是核心，不要泛泛而谈。
- 输出必须匹配现有 JSON schema。

核心指令：

```text
你是最终主编。DeepSeek 的研究稿只是参考，不是结论。
你需要重新判断哪些主题真正值得进入 keyDevelopments。
每条关键变化必须有事实依据、重要性解释和明确编辑判断。
不要为了覆盖素材而凑条目。
```

## 8. 成本控制策略

### 8.1 控制 Claude 输入

Claude 成本控制的关键不是少输出，而是少输入。建议：

- Claude 不读取所有 items。
- Claude 只读取 DeepSeek 聚类后的候选素材。
- 每个 cluster 限制 facts 数量。
- 每条原始素材只给标题、摘要、URL、短摘录。
- `briefCandidates` 最多 6-10 条。
- `dropCandidates` 不传全文，只传数量和原因摘要。

### 8.2 分层截断

建议使用以下默认限制：

- fetch 后最多 `filter.maxItems` 条，现有逻辑保留。
- DeepSeek 输入最多 80 条。
- 每条 DeepSeek 输入摘录最多 1000 字符。
- DeepSeek 输出最多 12 个 clusters。
- Claude 输入最多 35 条原始素材。
- Claude 最终 `keyDevelopments` 仍按 editorial 配置限制，例如 high 5 条、medium 3 条。

### 8.3 成本预算保护

二期可增加预算保护：

```yaml
llmPipeline:
  budget:
    maxWriterInputTokens: 20000
    maxEstimatedCostCny: 3.0
    onExceeded: compressMore # fail | compressMore | singleModel
```

第一版可以先不做预算硬中断，但审计必须把每阶段 token 和估算成本记清楚。

## 9. 失败回退策略

### 9.1 DeepSeek 预处理失败

可选策略：

- 默认：回退到当前单模型总结，避免任务完全失败。
- 如果 `llmPipeline.hybridRequired: true`，则直接报错。

推荐第一版默认回退，因为当前项目强调“单源失败不中断、健壮性优先”。

### 9.2 Claude 最终成稿失败

Claude 阶段失败时有两种选择：

- 使用 DeepSeek 根据中间稿生成最终 report，质量下降但可用。
- 直接报错，避免输出质量不达标的简报。

推荐提供配置：

```yaml
llmPipeline:
  writer:
    onFailure: fallbackToPreprocessModel # fail | fallbackToPreprocessModel
```

如果这是正式每日简报，建议默认 `fail`，因为 Claude 成稿失败后用 DeepSeek 兜底可能违背“最终观点交给 Claude”的目标。

### 9.3 JSON 解析失败

沿用现有 `repairAndParseJson`，但多阶段要在错误信息里带上 stage：

```text
LLM JSON parse failed at stage=final_write model=claude-...
```

## 10. 实施步骤

### 阶段一：多模型调用层

目标：不改变业务行为，只让 `llm.js` 支持 role。

改动：

- 扩展 `src/llm.js`。
- `.env.example` 增加多模型变量。
- 审计日志记录 `role` 和 `stage`。
- 单元测试覆盖默认回退和 role 配置读取。

验收：

- 未配置 `LLM_PREPROCESS_*` / `LLM_WRITER_*` 时，现有 `npm run brief` 行为不变。
- 配置不同 role 后，审计日志能看到不同模型名。

### 阶段二：DeepSeek 预处理

目标：新增中间稿，但暂不改变最终成文模型。

改动：

- 新增 `src/summarize/preprocess.js`。
- 定义中间 JSON schema。
- 审计中记录预处理输入条数、输出 cluster 数、drop 数。

验收：

- DeepSeek 能稳定输出可解析中间 JSON。
- 中间稿明显短于原始素材。
- drop/brief/cluster 建议可读、可追踪到 itemId。

### 阶段三：Claude 最终成稿

目标：启用 `mode: hybrid` 后由 Claude 生成最终 report。

改动：

- 新增 `src/summarize/write-report.js`。
- `src/summarize.js` 改为 single/hybrid 编排。
- Claude prompt 接入 editorial 和 DeepSeek 中间稿。

验收：

- 输出 JSON 与 `output.js` 兼容。
- Markdown 结构不变。
- keyDevelopments 的选题和 editorTake 质量明显优于单 DeepSeek。
- Claude 输入 token 明显低于“把所有 items 直接交给 Claude”。

### 阶段四：审计与成本分析

目标：让用户能判断多模型融合是否真的省钱。

改动：

- `auditor.js` 汇总 `llm.stages`。
- audit CLI 展示每阶段 token、模型、成本、耗时。
- 成本价格支持 role 级配置。

验收：

- audit summary 可以回答：
  - DeepSeek 预处理花了多少。
  - Claude 成稿花了多少。
  - 总成本比纯 Claude 低多少。
  - 总耗时是否可接受。

### 阶段五：可选校验

目标：提升稳定性，但不阻塞第一版。

改动：

- 新增 validator prompt。
- 检查最终 report 是否 schema 完整、是否疑似编造、是否违反 excludeTopics。
- 对轻微问题可自动修复，严重问题报错。

验收：

- 能捕捉空字段、markdown 围栏、错 schema、明显越界内容。

### 11.1 单元测试

建议新增或扩展：

- `src/llm.test.js`
  - role 环境变量读取。
  - 缺失 role 配置时回退默认模型。
  - price 配置读取。

- `src/summarize/preprocess.test.js`
  - items 摘录长度限制。
  - itemId 保持稳定。
  - 中间 JSON 基本校验。

- `src/summarize/write-report.test.js`
  - Claude prompt 包含 editorial、clusters、原始证据。
  - 输出 schema 与现有 report 字段一致。

- `src/utils/auditor.test.js`
  - 多阶段 llm 事件汇总。
  - 成本按 stage 汇总。

### 11.2 回归测试

- `llmPipeline` 不配置时，旧测试应全部通过。
- `mode: single` 时，行为与当前一致。
- `mode: hybrid` 时，可用 mock LLM 返回固定 JSON，验证编排顺序和输出。

### 11.3 人工质量评估

同一批新闻跑三份：

1. 当前 DeepSeek 单模型。
2. 纯 Claude 单模型。
3. DeepSeek + Claude hybrid。

对比指标：

- 关键选题是否更准。
- 是否减少弱相关新闻进入 keyDevelopments。
- editorTake 是否更有判断。
- editorReview 是否少空话。
- 成本是否显著低于纯 Claude。
- 输出耗时是否可接受。

## 12. 推荐默认方案

第一版建议采用以下默认配置：

```yaml
llmPipeline:
  mode: hybrid
  preprocess:
    provider: preprocess
    maxInputItems: 80
    itemExcerptChars: 1000
    maxClusters: 12
    maxFactsPerCluster: 5
  writer:
    provider: writer
    maxSourceItems: 35
    requireOriginalEvidence: true
    onFailure: fail
  validation:
    enabled: false
```

对应模型：

- DeepSeek：`preprocess`，低温度 `0.2`，偏结构化。
- Claude：`writer`，温度 `0.6-0.8`，偏最终表达。

## 13. 关键取舍

### 为什么不让 DeepSeek 直接决定最终选题？

因为“哪些新闻值得展开”不是纯整理任务，而是编辑判断。用户明确希望观点、选题和最终成文交给 Claude，因此 DeepSeek 只做候选和证据整理，Claude 保留最终裁量权。

### 为什么 Claude 还需要看原始素材？

如果 Claude 只看 DeepSeek 中间稿，它会被 DeepSeek 的判断框住，且更容易继承误判。给 Claude 精简后的原始标题、摘要、URL 和摘录，可以让它重新判断，同时不至于成本失控。

### 为什么不第一版就做三模型/多轮辩论？

多轮辩论会显著增加复杂度、成本和不可控性。当前最核心的问题是“低成本处理大素材 + 高质量最终表达”，两段式已经能解决主要矛盾。校验阶段可以等主流程稳定后再加。

## 14. 最终结论

推荐把新闻简报生成从“单模型直接成稿”升级为“研究助理 + 主编”的两段式架构：

- DeepSeek 是研究助理：便宜、大吞吐、结构化、负责压缩和候选整理。
- Claude 是最终主编：负责选题、观点、排序、表达和最终 JSON。

这套方案能在成本和质量之间取得更好的平衡：Claude 不再被迫阅读全部原始噪声，但仍能掌握足够证据做最终判断；DeepSeek 的优势被用在大批量整理上，而不是承担它相对较弱的最终写作和独立观点输出。

第一版应优先实现多模型调用层、DeepSeek 预处理、Claude 成稿和阶段级审计。校验、预算硬限制、自动压缩重试可以作为二期增强。
