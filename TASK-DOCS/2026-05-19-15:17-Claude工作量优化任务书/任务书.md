# Claude 工作量优化任务书

## 一、任务背景

当前 `global-geopolitical-conflicts-rss` 已启用多模型融合：

- DeepSeek 负责预处理新闻素材；
- Claude 负责最终成稿。

最近一次运行中，DeepSeek 预处理约 102 秒，Claude 最终成稿约 687 秒。Claude 阶段输入约 26k tokens，输出约 4.5k tokens，耗时明显偏高。

当前问题不是 Claude 不能调用，而是 Claude 阶段承担了过多“二次研究”和“重新筛选”工作。它既要读 DeepSeek 的聚类结果，又要读大量原始素材，还要重新判断选题、排序和成稿，导致耗时和成本都偏高。

## 二、任务目标

将多模型协作逻辑调整为：

> DeepSeek 读取原始新闻，完成筛选、压缩、聚合和准成稿编辑包；Claude 只做最后主编定稿。

核心目标：

- Claude 不再阅读大批量原始素材；
- Claude 不再承担完整二次研究；
- Claude 输入与输出规模控制在最多 4:1；
- 如果期望 Claude 输出约 2000 字，则 DeepSeek 交给 Claude 的内容最多约 8000 字；
- 最终质量仍保持“私人内参编辑”风格，有判断、有排序、有 editorTake。

## 三、职责边界

### DeepSeek 职责

DeepSeek 是“研究助理 + 案头主任”，负责重工作：

- 阅读过滤后的原始新闻素材；
- 去重、合并同类报道；
- 剔除弱相关、重复、体育娱乐、低价值素材；
- 判断哪些新闻值得展开；
- 为每条主线提取 2-3 条关键事实；
- 每条事实保留来源、URL、itemId；
- 输出可直接交给 Claude 使用的 `editorialPacket`；
- 控制 `editorialPacket` 总长度，默认不超过 8000 中文字。

### Claude 职责

Claude 是“最终主编”，只负责最后工作：

- 基于 DeepSeek 的 `editorialPacket` 成稿；
- 优化中文表达、标题、排序、语气；
- 强化 `editorTake`；
- 保持 JSON schema 兼容现有 `output.js`；
- 不新增事实；
- 不重新扩展素材范围；
- 不读取大批量原始素材。

## 四、目标流程

### 阶段 1：抓取与过滤

保持现有流程：

1. RSS 抓取；
2. 时间过滤；
3. 关键词过滤；
4. 排除关键词过滤；
5. URL 去重；
6. 截断到 `filter.maxItems`。

输出：进入模型链路的 `items`，例如 60 条。

### 阶段 2：DeepSeek 生成编辑包

DeepSeek 输入：

- 过滤后的新闻素材；
- 标题、摘要、正文摘录、来源、URL、发布时间；
- editorial 配置中的 persona、tone、interests、excludeTopics。

DeepSeek 输出新的中间结构 `editorialPacket`：

```json
{
  "meta": {
    "sourceItemCount": 60,
    "selectedItemCount": 18,
    "droppedItemCount": 42,

  "recommendedOrder": ["主线1", "主线2"],
  "keyDevelopmentsDraft": [
    {
      "title": "短标题",
      "importance": "high",
      "whatHappened": "事实草稿",
      "whyItMatters": "重要性草稿",
      "suggestedEditorTake": "给 Claude 的观点建议",
      "evidence": [
        {
          "itemId": 1,
          "source": "Reuters",
          "url": "https://...",
          "fact": "关键事实"
        }
      ]
    }

  "briefsDraft": [
    {
      "region": "地区/领域",
```

### 阶段 3：Claude 最终定稿

Claude 输入：

- `editorialPacket`；
- 最小必要的 schema 要求；
- editorial 语气要求；
- 不再默认附带 35 条原始新闻素材。

Claude 输出：

- 现有 `output.js` 兼容的最终 report JSON：
  - `tldr`
  - `overview`
  - `keyDevelopments`
  - `briefs`
  - `timeline`
  - `signals`
  - `risks`
  - `unknowns`
  - `editorReview`

Claude 约束：

- 只能使用 `editorialPacket` 中的事实；
- 不新增来源、数字、人物、时间、地点；
- 可以调整表达、标题和排序；
- 可以压缩或合并条目；
- 不做大范围事实重筛。

## 五、配置设计

建议在 `llmPipeline` 中新增或调整配置：

```yaml
llmPipeline:
  mode: hybrid

  preprocess:
    maxInputItems: 80
    itemExcerptChars: 700
    maxDevelopments: 6
    maxFactsPerDevelopment: 3
    maxBriefs: 6
    maxPacketChars: 8000
    outputMode: editorialPacket

  writer:
    inputMode: editorialPacket
    includeRawSourceItems: false
    maxInputOutputRatio: 4
    targetOutputChars: 2000
    onFailure: fail
```

配置含义：

- `maxPacketChars`：DeepSeek 给 Claude 的最大中文字符数；
- `inputMode: editorialPacket`：Claude 只读取压缩编辑包；
- `includeRawSourceItems: false`：默认不再附加原始素材；
- `maxInputOutputRatio: 4`：控制 Claude 输入/输出目标比例；
- `targetOutputChars`：指导 Claude 控制最终内容长度。

## 七、验收标准

功能验收：

- `global-geopolitical-conflicts-rss` 能完整跑通；
- 审计日志中出现 DeepSeek `editorialPacket` 相关事件；
- Claude writer 阶段输入显著下降；
- Claude 不再默认接收 35 条原始素材；
- 最终输出 Markdown 和 JSON 结构不变。

性能验收：

- Claude 输入/输出规模不超过 4:1 的目标区间；
- 以 2000 字左右最终输出为目标，Claude 输入控制在约 8000 字以内；
- Claude 阶段耗时相比当前 11 分钟明显下降，目标控制在 3-6 分钟内。

质量验收：

- 最终简报仍有明确主线；
- keyDevelopments 不堆砌事实；
- editorTake 保持有判断、有立场；
- 不出现 Claude 编造 DeepSeek 未提供的事实；
- briefs、timeline、signals、risks、unknowns 字段仍然可用。

日志验收：

- 审计日志记录：
  - DeepSeek 输入条数；
  - DeepSeek packet 字符数；
  - Claude 输入字符数；
  - Claude 输出 token；
  - Claude 耗时；
  - 是否附带原始素材；
  - 是否触发 fallback。

## 八、风险与处理

### 风险 1：DeepSeek 压缩过度，Claude 缺事实

处理：

- 每条 keyDevelopment 至少保留 2 条证据；
- 每条证据保留 URL；
- 对 high importance 主线允许更多证据。

### 风险 2：Claude 变成纯润色，判断力下降

处理：

- DeepSeek 输出 `suggestedEditorTake`，但 Claude 可以重写；
- Claude 仍可调整排序、合并条目、强化观点；
- Claude 不负责事实重筛，但负责表达和最终判断呈现。

### 风险 3：DeepSeek 输出 JSON 不稳定

处理：

- 加强 `validatePreprocessResult`；
- 对缺字段给默认值；
- 必要时增加一次轻量 repair；
- 失败时按配置决定 fail 或 fallback。

### 风险 4：8000 字限制影响覆盖面

处理：

- 保留 `maxPacketChars` 可配置；
- 正式主题先设 8000；
- 如质量不足，可调到 10000 或减少原始输入条数。

## 九、建议实施顺序

1. 修改 DeepSeek prompt，产出 `editorialPacket`；
2. 修改校验逻辑，保证 packet 字段稳定；
3. 修改 Claude writer prompt，只做最终定稿；
4. 修改正式主题配置；
5. 补审计日志字段；
6. 跑 `test-hybrid` 小样本验证；
7. 跑 `global-geopolitical-conflicts-rss` 正式验证；
8. 对比优化前后的 Claude 输入、耗时、输出质量。

## 十、预期效果

优化前：

- DeepSeek 做预处理；
- Claude 读完整中间稿 + 35 条素材；
- Claude 输入约 26k tokens；
- Claude 阶段耗时约 11 分钟。

优化后目标：

- DeepSeek 输出 8000 字以内编辑包；
- Claude 只读编辑包；
- Claude 输入降到约 8k-12k tokens；
- Claude 阶段目标耗时 3-6 分钟；
- 输出质量保持或提升，因为 Claude 注意力集中在表达和最终编辑，而不是重复研究。
