# 新闻简报工作流 — 流程介绍

## 一句话说清楚这个项目

> 定时从 RSS 新闻源抓取最新报道，用大模型（LLM）自动生成结构化的中文简报，输出到 Obsidian 笔记库。

## 运行流程（4个步骤）

### 步骤 1：加载配置 (`src/config.js`)

```
用户执行：npm run brief us-iran
                │
                ▼
     读取 config/topics/us-iran.yaml
                │
                ▼
     解析成 JS 对象 + 校验必填字段
```

配置文件长什么样？以 `us-iran.yaml` 为例：

```yaml
id: us-iran                              # 主题唯一ID
title: 美国伊朗局势速报                    # 简报标题（中文）

sources:                                 # 新闻来源列表
  - name: Al Jazeera                     #   源名称
    url: "https://..."                   #   RSS地址

filter:                                  # 过滤规则
  keywords: [Iran, sanctions, ...]       #   命中任意关键词才保留
  lookbackHours: 36                      #   只看最近36小时的新闻
  maxItems: 40                           #   最多保留40条

output:                                  # 输出设置
  dir: "/Users/.../ObsidianWithAI/..."   #   简报输出目录
```

**设计要点**：加新主题只需新建一个 YAML 文件，代码一行不改。

---

### 步骤 2：抓取与过滤 (`src/fetch.js`)

这是数据获取层，像一个多级筛子：

```
RSS源1 ─┐
RSS源2 ─┼─ 并发拉取
RSS源3 ─┘
    │
    ▼
┌─────────────────┐
│ 筛子1: 时间过滤   │ → 丢弃36小时前的旧闻
├─────────────────┤
│ 筛子2: 关键词过滤 │ → 标题/摘要不含关键词的丢掉
├─────────────────┤
│ 筛子3: URL去重   │ → 同一篇文章只留第一次出现
├─────────────────┤
│ 筛子4: 截断      │ → 按时间倒序，只取前40条
└─────────────────┘
    │
    ▼
  40条以内的高质量新闻条目
```

**关键的容错设计**：如果某个 RSS 源挂了（网络超时、被封），它返回空数组，不影响其他源的数据。不会因为一个源出问题就让整个流程崩溃。

**数据统一化**：不管 RSS 源是什么格式（字段名可能不同），全部转为统一结构：

```js
{
  title: "新闻标题",
  url: "https://...",          // 原文链接
  source: "Al Jazeera",        // 来源名称
  publishedAt: "2026-05-05T...",
  summary: "新闻摘要文本..."
}
```

---

### 步骤 3：LLM 总结 (`src/summarize.js` + `src/llm.js`)

这是核心的"智能"环节。两个文件分工明确：

| 文件 | 职责 |
|------|------|
| `src/llm.js` | 封装 LangChain，提供通用的 `callLLMForJson()` 接口 |
| `src/summarize.js` | 写具体的 prompt（提示词），定义输出格式 |

**工作流程**：

```
40条新闻条目
    │
    ▼
summarize.js 把新闻拼成一段大文本
    │
    ├── SYSTEM PROMPT: "你是资深国际新闻编辑，只基于素材，中文输出..."
    │
    ├── USER PROMPT:  "请基于40条新闻生成简报，返回JSON格式..."
    │   （里面包含了全部40条新闻的标题、摘要、来源）
    │
    ▼
llm.js 的 callLLMForJson()
    │
    ├── ChatPromptTemplate（组装system+user消息）
    ├── ChatOpenAI（调用 DeepSeek / OpenAI 等模型API）
    └── JsonOutputParser（把模型返回的JSON字符串解析为JS对象）
    │
    ▼
{
  summary: "一句话概览,40字以内",
  keyDevelopments: ["关键变化1", "关键变化2", ...],
  timeline: [{time: "05-05 14:30", event: "..."}, ...],
  risks: ["风险观察..."],
  unknowns: ["信息缺口..."]
}
```

**关于 `src/llm.js` —— 为什么单独抽出来？**

`llm.js` 是这个项目最重要的扩展点。它是唯一接触 LangChain SDK 的地方。未来你想：
- 换模型（DeepSeek → GPT → 通义千问）→ 改 `.env` 里的环境变量
- 加重试（调用失败自动重试3次）→ 只改 `llm.js`
- 加日志（记录每次调用的 token 消耗）→ 只改 `llm.js`
- 多模型 fallback（主模型挂了切备用模型）→ 只改 `llm.js`

`.env` 里只需要三个环境变量：

```
LLM_API_KEY=sk-xxx                              # API密钥
LLM_BASE_URL=https://api.deepseek.com/v1        # API地址
LLM_MODEL=deepseek-chat                         # 模型名称
```

换成其他服务商（通义千问、Moonshot、智谱GLM）只需改 `LLM_BASE_URL` 和 `LLM_MODEL`。

### 步骤 4：输出文件 (`src/output.js`)

同时生成两个文件：

**Markdown 文件** — 给人看的，有 formatting：

```
2026-05-05-美国伊朗局势速报.md
```

内容结构：
```markdown
---
topic: us-iran
title: 美国伊朗局势速报
date: 2026-05-05
generatedAt: 2026-05-05T08:00:00.000Z
sourceCount: 24
---

# 美国伊朗局势速报

## 一句话概览
今天美伊关系出现重大转折...

## 关键变化
1. 变化一
2. 变化二

## 时间线
- 05-05 08:20 事件A
- 05-05 09:10 事件B

## 风险与观察
- 风险描述...

## 信息缺口
- 缺乏某方面的报道...

## 来源
- [Al Jazeera: 新闻标题](https://...)
```

**JSON 文件** — 给程序读的，保留了全部结构化数据：

```json
{
  "topic": "us-iran",
  "title": "美国伊朗局势速报",
  "date": "2026-05-05",
  "summary": "...",
  "keyDevelopments": ["...", "..."],
  "timeline": [{"time": "05-05 08:20", "event": "..."}],
  "risks": ["..."],
  "unknowns": ["..."],
  "sources": [{ "title": "...", "url": "...", "source": "Al Jazeera", ... }]
}
```

**关键设计**：
- 文件名带日期 → 同一天多次运行会覆盖旧文件，自然的幂等性
- 字段缺失时显示"（无）" → LLM有时会偷懒不返回某个字段，不因此报错
- YAML frontmatter → Obsidian 能自动识别为笔记属性

## 文件职责速查表

| 文件 | 一句话职责 | 依赖 |
|------|-----------|------|
| `src/index.js` | CLI 入口，串联4步流程 | 下面4个模块 |
| `src/config.js` | 读 YAML 配置 + 校验 | `yaml` 包 |
| `src/fetch.js` | 拉 RSS + 时间/关键词/去重/截断 | `rss-parser` 包 |
| `src/llm.js` | LangChain 封装，统一 LLM 调用 | `@langchain/openai` |
| `src/summarize.js` | 写 prompt，调 LLM 生成简报 | `src/llm.js` |
| `src/output.js` | 写 .md 和 .json 到 Obsidian | 无 |
| `config/topics/*.yaml` | 主题配置（源、关键词、输出） | 被 `config.js` 读取 |
| `.env` | LLM 密钥和地址 | 被 `llm.js` 使用 |

```

    │
    ▼
┌──────────────┐    HTTP请求     ┌─────────┐

│    + llm.js  │ ◀────────────── │(DeepSeek)│
└──────┬───────┘    JSON响应     └─────────┘
       │
       │ 结构化简报 (JSON object)
       ▼
┌──────────────┐    文件写入     ┌──────────────┐
│  output.js   │ ──────────────▶ │ Obsidian库   │
│              │                 │ .md + .json  │
└──────────────┘                 └──────────────┘
```

## 扩展指南

作为新手，未来你想加新功能时应该改哪里：

| 想做什么 | 改哪个文件 | 说明 |
|----------|-----------|------|
| 加新主题（如"AI行业动态"） | 新建 `config/topics/ai.yaml` | 代码不动 |
| 给现有主题加RSS源 | 改对应的 `.yaml` 文件 | 代码不动 |
| 换 LLM 服务商 | 改 `.env` 里的环境变量 | 代码不动 |
| 调 prompt，改输出风格 | `src/summarize.js` | 改提示词文本 |
| 改过滤规则（如只看24小时） | 改 `.yaml` 里的 `lookbackHours` | 代码不动 |
| 加 API 类型的新闻源 | `src/fetch.js` | 加新函数，yaml里加type字段 |
| 加重试机制 | `src/llm.js` | 在 callLLMForJson 里加 |
| 加运行日志 | `src/index.js` 或新建文件 | 记录中间结果 |

## 和 OpenClaw 的关系

这个项目本身只负责"生成简报"。外部调度（定时触发）和推送（发飞书）由另一套系统 OpenClaw 负责。两者通过**文件系统**通信：

```
OpenClaw cron  ──触发──▶  npm run brief us-iran  ──产出──▶  .md 文件
                                                              │
OpenClaw 读取最新 .md ◀───────────────────────────────────────┘
    │
    └──推送──▶  飞书群
```

这个解耦设计的好处：
- 新闻工作流可以独立开发、测试、运行
- 换推送渠道（飞书 → 钉钉 → 邮件）不需要改新闻工作流
- 即使推送系统挂了，简报照样生成，不会丢数据
