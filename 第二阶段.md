# Phase 2.5:通用 AI 网页抓取适配器

> 本任务书追加在 `PLAN.md` 的 Phase 2 之后,Phase 3 之前。
> 在 Phase 2 已合并、人类 review 通过后启动。
> 全局开发约束(代码风格、依赖管理、兼容性、测试要求、提交规范、不要做的事)沿用 PLAN.md。

---

## 背景

Phase 2 实现了 4 种 source type:`rss` / `html` / `api` / `playwright`,但 `html` 和 `playwright` 类型仍需要手写 CSS 选择器。
对中等以上规模的源(20+),维护选择器是显著负担:

- 配新源要 F12 找元素
- 网站改版会静默失败(选择器选不出东西)
- 反爬严的站(基础 UA/headless 检测)用 `html` 类型直接被拒
- 需要登录的内部系统当前无法处理

## Phase 2.5 目标

**新增一种 source type:`web`,实现"给一个 URL,自动搞定剩下的事"**。
设计取舍明确接受:**牺牲速度和成本,换心智简化**。

- 新源配置只需 `name` + `url` 两个字段
- 浏览器抓取(应对 JS 渲染、基础反爬)
- AI 提取列表和详情(应对结构差异、网站改版)
- 全程显式步骤打印(用户能实时看到进度)
- 严格日志长度限制(不把抓到的页面内容打满终端)
- 多源并发受控(避免源多时浏览器/内存爆掉)
- 登录场景留接口,但不在本 Phase 实现完整登录流程

## 不在本 Phase 范围内

明确**不做**的事,避免范围蔓延:

- ❌ 复杂登录流程的可视化配置(留 `sessionFile` 字段供后续扩展)
- ❌ 验证码识别 / Cloudflare 高级反爬
- ❌ 学习 selector + 缓存机制(那是另一个方向,跟本 Phase 是不同取舍)
- ❌ 替换或修改 Phase 2 已有的 `html`/`playwright` 适配器(它们继续保留供老配置使用)
- ❌ AI 临场决策"下一步点哪里"(不可控,本 Phase 不做)

## 设计原则(本 Phase 特有)

1. **一种 type 通吃**:用户不需要在多种 type 间做选择题,所有非 RSS 场景统一走 `type: web`
2. **浏览器 + AI 是标配**:不为性能做特殊优化路径(如果未来要优化,作为单独 Phase)
3. **失败要响亮**:任何步骤失败都明确报错,不要"半成功"地往下走
4. **日志是一等公民**:每个关键步骤打印,且每条日志严格限长
5. **AI 调用集中在一处**:所有 prompt 集中管理,不分散到各模块

## 2.5.1.A 步骤化日志器

### 模块:`src/utils/logger.js`

### 设计目标
- 给所有抓取流程提供统一的步骤打印
- 强制限制每条日志长度,避免把网页/JSON 内容刷屏
- 带时间戳和上下文标签(哪个 source、哪一步)
- 终端友好:用 ANSI 颜色区分级别(info/warn/error/step)

### 接口

```js
/**
 * 创建一个带上下文的 logger 实例
 * @param {string} context - 上下文标签,如 "web/BBC News"
 * @returns {Logger}
 */
export function createLogger(context) { ... }

// Logger 实例方法:
logger.step(stepName, detail)    // 打印一个步骤,如 "step: 打开页面 [https://bbc.com]"
logger.info(message, detail)     // 普通信息
logger.warn(message, detail)     // 警告
logger.error(message, detail)    // 错误
logger.success(message, detail)  // 成功
logger.timing(label, ms)         // 耗时统计

// detail 是可选的附加信息,如:
//   logger.step('提取链接', { count: 12, sample: 'https://...' })
```

### 输出格式

```
[14:23:01] [web/BBC News]   ▶ 打开页面 https://bbc.com/news/world/middle_east
[14:23:04] [web/BBC News]   ✓ 页面加载完成 (2843ms, 187KB)
[14:23:04] [web/BBC News]   ▶ HTML 瘦身
[14:23:04] [web/BBC News]   ✓ 瘦身完成 (187KB → 18KB, 90% 压缩)
[14:23:04] [web/BBC News]   ▶ AI 提取链接 (model: deepseek-chat)
[14:23:09] [web/BBC News]   ✓ 提取到 12 个链接 (5234ms, 1842 tokens)
[14:23:09] [web/BBC News]   ▶ 抓取详情页 1/12: https://bbc.com/news/world-middle-east-12345
...
```

### 长度限制规则(硬性)

| 字段类型 | 最大长度 | 超长处理 |
|---------|---------|---------|
| URL | 80 字符 | 中间用 `...` 省略,保留协议+域名+末尾 |
| 文本预览(标题、片段) | 60 字符 | 末尾用 `...` 截断 |
| HTML/JSON 内容 | **永远不打印完整内容**,只打印长度统计 | — |
| Token 数 / 字节数 | 用 K/M 等单位 | — |
| 错误消息 | 200 字符 | 末尾截断 |
| 单条日志总长 | 250 字符 | 截断尾部 |

### 实现要点

```js
// 内部工具函数
function truncate(str, max, suffix = '...') {
  if (!str || str.length <= max) return str;
  return str.slice(0, max - suffix.length) + suffix;
}

function formatUrl(url, max = 80) {
  if (url.length <= max) return url;
  // 保留协议+域名+尾部路径,中间省略
  const u = new URL(url);
  const tail = u.pathname.slice(-30);
  return `${u.protocol}//${u.host}/...${tail}`;
}

function formatBytes(bytes) {
  if (bytes < 1024) return `${bytes}B`;
  if (bytes < 1024 * 1024) return `${(bytes / 1024).toFixed(1)}KB`;
  return `${(bytes / 1024 / 1024).toFixed(1)}MB`;
}
```

### detail 渲染规则

`detail` 是对象时,自动格式化为 `key=value` 串,但每个值也走截断:

```js
logger.step('AI 提取链接', { tokens: 1842, model: 'deepseek-chat', sample: 'https://...' })
// 输出: ▶ AI 提取链接 (tokens=1842, model=deepseek-chat, sample=https://...)
```

特殊 key 走专门格式化:
- `url` / `urls` → `formatUrl`
- `bytes` / `size` / `length` → `formatBytes`
- `ms` / `duration` → `${ms}ms`
- `tokens` → 整数显示,带千位逗号

### 全局开关

环境变量 `LOG_LEVEL` 控制最低输出级别:
- `verbose`:所有都打
- `info`(默认):info 及以上
- `warn`:只打警告和错误
- `quiet`:只打错误

### 测试

`src/utils/logger.test.js`:
- truncate / formatUrl / formatBytes 单测
- 模拟 stdout 输出,验证长度不超限
- 验证 LOG_LEVEL 过滤行为

## 2.5.1.B HTML 瘦身工具

### 模块:`src/utils/html-cleaner.js`

### 设计目标
把动辄几百 KB 的页面 HTML 压成 LLM 友好的小文本,降低 token 消耗,同时保留足够的结构信息让 AI 能理解页面布局。

### 接口

```js
/**
 * @param {string} html
 * @param {object} [options]
 * @param {'list' | 'article'} [options.mode='list']
 * @param {number} [options.maxChars=30000]
 * @returns {{ cleaned: string, originalLength: number, cleanedLength: number, truncated: boolean }}
 */
export function cleanHtml(html, options = {}) { ... }
```

### 必做的清理(无论 mode)

- 移除标签:`<script>`, `<style>`, `<link>`, `<meta>`, `<noscript>`, `<svg>`, `<iframe>`, `<canvas>`, `<picture>`, `<video>`, `<audio>`
- 移除 HTML 注释
- 移除所有元素的事件属性(`on*`)
- 移除 `style` 属性
- 移除 `data-*` 属性(99% 是垃圾)
- 移除 base64 内联资源(`data:image/...`、`data:font/...`)
- 折叠连续空白(多个空格/换行 → 单个空格)
- 移除空的 div/span(没有文本和子元素的)

### list 模式额外清理
- 移除明显的非内容区:`<header>`, `<footer>`, `<nav>`, `<aside>`
- 移除 `role="banner|navigation|complementary|contentinfo"` 元素
- **保留所有 `<a>` 标签的 `href` 和文本**(这是 AI 提取链接的依据)
- **保留 class/id**(但只保留前 50 字符,避免 utility-class 框架如 Tailwind 把 class 撑爆)

### article 模式额外清理
- 优先级查找:`<article>` > `<main>` > `[role=main]` > 文本量最大的 `<div>`
- 找到主内容区后,丢弃其余
- 不需要保留 class/id(详情页用 Readability 提取就够)

### 截断策略
- 超过 `maxChars` 时从尾部截断(列表通常在头部)
- `truncated: true` 时上层可以打 warn

### 实现要点
- 用 `cheerio` 操作 DOM(项目已有依赖,不引入新包)
- class 截断:`<div class="a b c d e f g ...">` → `<div class="a b c">` (只保留前 N 个)

### 测试

`src/utils/html-cleaner.test.js`:
- 准备 fixture(脱敏的真实新闻站 HTML)放在 `test/fixtures/cleaner/`
- 验证压缩率 > 70%
- 验证 `<a>` 标签数量保持
- 验证 script/style 完全消失
- 验证截断标记正确

# Phase 2.5.2:type: web 适配器

**目标**:实现新的 source type `web`,用 Playwright + AI 通用抓取。
**Branch**:`feature/phase-2.5.2-web-adapter`
**前置依赖**:Phase 2.5.1 已合并

## yaml schema

```yaml
sources:
  # 极简形态:这是常见配置
  - name: BBC 中东
    type: web
    url: "https://www.bbc.com/news/world/middle_east"

  # 完整形态(所有字段都可选)
  - name: 某复杂源
    type: web
    url: "https://example.com/news"
    hint: "新闻列表在主面板,忽略侧边推荐"   # 给 AI 的额外上下文
    maxArticles: 15                          # 默认 10
    waitFor:                                 # 加载等待策略
      type: networkidle                      # 'networkidle' | 'selector' | 'timeout'
      selector: ".news-loaded"               # type=selector 时用
      timeoutMs: 15000                       # 上限
    sessionFile: "state/sessions/example.json"  # 复用登录态(本 Phase 仅消费,不创建)
    fetchDetail: true                        # 默认 true,是否抓详情页
    detailExtraction: auto                   # 'auto' | 'readability' | 'ai'
                                             #   auto: 先 readability,失败再 AI
                                             #   readability: 只用 readability
                                             #   ai: 直接用 AI 提取
```

## 整体流程

```
yaml: { type: web, url: X }
        │
        ▼
[step 1] 浏览器打开 X (按 waitFor 策略等加载)
        │
        ▼
[step 2] 拿到完整渲染后的 HTML
        │
        ▼
[step 3] HTML 瘦身 (list 模式)
        │
        ▼
[step 4] AI 提取链接列表
        │   prompt: "这是新闻列表页 HTML,提取每条新闻的 {title, url, publishedAt?}"
        │   返回 JSON 数组
        ▼
[step 5] 链接合法化(相对路径 → 绝对、过滤无效链接)
        │
        ▼
[step 6] 截取前 maxArticles 条
        │
        ▼
[step 7] 对每条链接(并发受控):
         a. 浏览器打开
         b. 拿到 HTML
         c. 按 detailExtraction 策略提取正文
            - readability: 走现有 extractor
            - ai: 瘦身 article 模式 → AI 提取 {title, content, publishedAt, author}
            - auto: 先 readability,正文 < 100 字时降级到 ai
         d. 合并到 NewsItem
        │
        ▼
[step 8] 返回 NewsItem[]
```

## 关键实现细节

### 1. 浏览器复用

直接复用 Phase 2 `playwright.js` 的 browser 单例和 `shutdownPlaywright()` 钩子。
不要再起一个 browser。

每个 `fetchFromWeb` 调用使用独立的 context,失败时关 context,不关 browser。

### 2. 资源屏蔽

默认屏蔽图片/字体/CSS(继承 Phase 2 默认行为):

```js
await context.route('**/*.{png,jpg,jpeg,svg,gif,webp,woff,woff2,ttf,css,ico}',
  r => r.abort());
```

### 3. session 复用(被动消费)

如果 `sessionFile` 字段存在且文件存在,加载到 context:

```js
const sessionFile = source.sessionFile;
const contextOptions = {};
if (sessionFile && await fileExists(sessionFile)) {
  contextOptions.storageState = sessionFile;
  logger.info('加载登录态', { sessionFile });
}
const context = await browser.newContext(contextOptions);
```

session 文件本 Phase 不负责创建——后续可以用工具脚本(浏览器手动登录后保存),那是另一个 Phase 的事。

### 4. AI 提取列表(prompts.js)

system prompt(中文,严格):

```
你是一个网页内容提取助手。给你一段新闻列表页的 HTML,你需要识别其中所有新闻条目。

返回严格的 JSON 数组,不要 markdown 代码块,不要任何解释:
[
  {
    "title": "新闻标题",
    "url": "新闻详情页的相对或绝对 URL",
    "publishedAt": "ISO8601 时间或空字符串",
    "summary": "如果列表页有摘要则填,否则空字符串"
  }
]

约束:
- 只提取真正的新闻条目,排除导航、推荐、广告
- 如果识别不出 publishedAt,留空字符串(不要瞎猜)
- 如果一条都识别不出,返回空数组 []
- title 不要包含多余的"分类标签"前缀
```

user prompt:

```
URL: {currentUrl}
{hint ? `提示: ${hint}` : ''}

HTML:
{cleanedHtml}
```

### 5. URL 合法化

AI 返回的 url 可能是相对路径(`/article/123`)或绝对路径,统一处理:

```js
function normalizeUrl(rawUrl, baseUrl) {
  try {
    return new URL(rawUrl, baseUrl).href;
  } catch {
    return null;  // 无效 URL 丢弃
  }
}
```

过滤掉:
- 无法解析的 URL
- 协议不是 http/https 的(`javascript:`, `mailto:`, `tel:`)
- 跟 baseUrl 不同域且明显不像新闻详情的(可选,默认不做)

### 6. AI 提取详情(prompts.js)

system prompt:

```
你是一个新闻正文提取助手。给你一段新闻详情页的 HTML,提取以下字段:

返回严格的 JSON 对象,不要 markdown,不要解释:
{
  "title": "文章标题",
  "content": "完整正文(纯文本,保留段落分隔)",
  "publishedAt": "ISO8601 时间或空字符串",
  "author": "作者或空字符串"
}

约束:
- content 不要包含相关推荐、广告、评论区内容
- 段落之间用换行符分隔
- 如果某字段不确定,留空字符串
```

### 7. 详情页提取策略

```js
async function extractDetail(html, url, strategy) {
  const log = createLogger(`web/detail`);

  if (strategy === 'readability') {
    return await extractByReadability(html, url);
  }

  if (strategy === 'ai') {
    return await extractByAi(html, url);
  }

  // auto: 先 readability
  log.step('尝试 Readability');
  const r = await extractByReadability(html, url);
  if (r && r.content && r.content.length >= 100) {
    log.success('Readability 成功', { length: r.content.length });
    return r;
  }

  log.warn('Readability 内容不足,降级到 AI', { length: r?.content?.length ?? 0 });
  return await extractByAi(html, url);
}
```

### 8. 详情页并发

详情页抓取在单个 source 内部用 `p-limit` 控制(默认 2,可配):

```yaml
- name: ...
  type: web
  url: ...
  detailConcurrency: 3   # 默认 2
```

每个详情页单独打 context 还是复用?**复用**——同一个 source 的详情页用同一个 context,登录态自然带上,也省资源。
失败时只关页面(page.close()),不关 context。

只能打印**长度、统计、URL、URL 摘要、时间**这类元信息。

### 10. 失败处理

| 失败场景 | 处理 |
|---------|------|
| 列表页加载超时 | 整个 source 失败,返回 [],打 error |
| AI 提取列表失败/返回空 | 整个 source 失败,返回 [],打 error |
| AI 返回 JSON 格式错 | 重试一次,仍失败则放弃这个 source |
| 单个详情页抓取失败 | 跳过该详情,继续下一条,打 warn |
| 详情页 ai 提取失败 | 用列表页的 title + url 兜底,content 留空 |

任何 AI 调用都用 `p-retry`,沿用 Phase 2 的重试设置。

### 11. 失败截图(可选)

列表页加载失败时截图到 `logs/web-failures/${timestamp}-${source}.png`,方便事后看。

# Phase 2.5.3:并发调度增强

**目标**:支持源数较多(20-50+)的场景,避免浏览器/内存爆掉。
**Branch**:`feature/phase-2.5.3-concurrency`
**前置依赖**:Phase 2.5.2 已合并

## 问题

当前 `fetch/index.js` 用单一 `p-limit` 控制全局并发。这在源类型混杂时不够好:

- RSS 源轻量,可以高并发
- web 源重(浏览器 + AI 调用),并发要严格限制
- 用统一上限会浪费 RSS 性能或撑爆 web 资源

## 方案:**按 type 分类的并发池**

每种 type 一个独立的 p-limit:

```yaml
filter:
  runtime:
    concurrency:
      rss: 8
      html: 5
      api: 5
      playwright: 2
      web: 2          # 新加
      default: 5
```

老格式 `concurrency: 5` 仍兼容(等价于把所有 type 的并发上限都设为 5)。

## 实现

`src/fetch/index.js`:

```js
function createPoolMap(concurrencyConfig) {
  if (typeof concurrencyConfig === 'number') {
    // 老格式:所有 type 共享同一个 pool
    const limit = pLimit(concurrencyConfig);
    return new Proxy({}, { get: () => limit });
  }
  // 新格式:每个 type 一个 pool
  const pools = {};
  const defaultLimit = pLimit(concurrencyConfig.default ?? 5);
  return new Proxy(pools, {
    get(target, type) {
      if (!target[type]) {
        const n = concurrencyConfig[type] ?? concurrencyConfig.default ?? 5;
        target[type] = pLimit(n);
      }
      return target[type];
    }
  });
}

// 调度时:
const pools = createPoolMap(filterConfig.runtime?.concurrency);
const results = await Promise.all(
  sources.map(s => {
    const type = s.type ?? 'rss';
    const pool = pools[type];
    return pool(() => fetchOne(s));
  })
);
```

# 给 codex 的特别提醒(Phase 2.5)

## 容易踩的坑

1. **HTML 瘦身做不彻底,AI 调用 token 超标** → 每次 AI 调用前必须打印瘦身后的字符数,验证 < 30K
2. **日志里不小心打了完整 HTML/正文** → review 每一处 logger 调用,确认只打元信息
3. **detail 并发开太大,浏览器 OOM** → 默认 2,不要轻易改大
4. **AI 返回的 JSON 格式错没兜底** → 必须用 try/catch 包 JSON 解析
5. **session 文件被 commit 进 git** → 先改 `.gitignore` 再开发
6. **失败时没关 context,内存泄漏** → finally 块关 context
7. **timeout 写得短了误杀正常请求** → 测试时观察实际耗时再定值

## 何时停下来问人类

- AI 提取的 prompt 措辞拿不准时(prompt 工程影响很大)
- 日志格式视觉不好看时(可以贴几行示例问)
- 浏览器资源屏蔽规则疑似导致漏抓时
- 源超时默认值不确定时
- 任何"这个错误算严重还是 warn"的判断模糊时
